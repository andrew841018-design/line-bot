✅ Pending 偷塞 piggyback — 免費回補機制 main.py _reply() ▼
  • 背景：LINE push_message 每月 200 則上限，4月已用完（HTTP 429）
  • 解法：每次有人觸發真實回應時，同時把最多 5 則 pending 回應夾帶進同一個 reply_message（LINE 允許 1 次 reply 最多 5 則訊息）
  • piggyback 格式：
    📬 補回之前漏掉的訊息
    原文：（原始訊息前 300 字）
    回應：（LLM 生成的回應）
  • 成功後從 pending_explicit_reply.json 移除已處理項目
  • LLM 失敗（exception）時靜默跳過，不影響主要回應
  • pending 只含真實使用者訊息（__bot__ 條目已過濾）
每次 reply 夾帶 1 批 pending；pending 逐漸清空；主回應不受影響

FLOW 訊息處理流程
📊 一則訊息進來的完整路徑 ▼
  • LINE webhook → POST /callback → 驗簽名 → 取出 events
  • 判斷類型：圖片/影片/貼圖 → 忽略；YouTube URL → yt-dlp 抓字幕；一般文字 → 繼續
  • 是否 @mention Bot？ → 是：直接呼叫 _handle_explicit_text()（立即回應） → 否：burst_filter.add_to_burst()（等 burst 結束）
  • burst_filter 計時結束 → _handle_explicit_text() → LLM waterfall
  • LLM waterfall：Gemini OK → 回應；Gemini 429 → Grok；Grok 也 429 → 寫 pending
  • 回應時呼叫 _reply() → 同時 piggyback 最多 5 則 pending → LINE reply_message
  • 寫入 memory DB → bot_stats.increment → 完成
本地測試：pkill -f "uvicorn main:app" && nohup .venv/bin/uvicorn main:app --host 127.0.0.1 --port 8080 > /tmp/line_bot_restart.log 2>&1 & 後 curl 發假 webhook
📦 Pending 群組分組流程（startup 時處理積壓） ▼
  • Bot 重啟時：讀取 pending JSON → 用 _gemini_group_messages() 分組（哪些訊息是同一對話串）
  • Gemini grouping 失敗 → grok_client.group_messages() → 還是失敗 → heuristic（每則各自）
  • 分組後用 push_message 批次推送（需要 LINE push 額度）
  • 目前 push 額度耗盡（HTTP 429）→ startup pending 推送失敗 → 改走 piggyback 機制
  • __bot__ 條目在處理前過濾（items = [it for it in items if it.get("user_id") != "__bot__"]）
